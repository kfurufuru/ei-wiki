# PLC/DCS トラブル

## 30秒まとめ

PLC/DCS トラブルは「物理層 → データリンク層 → 上位（コントローラ・ソフト）」の順に、**必ず実測で1段ずつ潰す**。ping が通らない・LED が赤いで止めず、端子電圧・チャネル LED・オンラインモニタまで降りて切り分ける。冗長構成では「今どちらが現用か」を最初に確定させる。プログラム変更を伴う対処は MOC（変更管理）手続きが必須で、SIS 絡みはさらに厳格な手続きが要る。

!!! tip "切り分けの原則：現物 → 表示 → ソフトの順で突き合わせる"
    「DCS 画面の値」「I/O カードの LED」「フィールド端子の実測値」の3つを突き合わせれば、
    異常が現場側・ハード側・ソフト側のどこにあるかがほぼ一意に絞れます。
    どれか1つだけを見て判断しないことが、最短で原因に届くコツです。

## 通信断の切り分け

通信断は「上位のコマンドが通らない＝ネットワーク故障」と早合点しがちですが、**物理層 → データリンク層 → 上位層**の3階層を下から潰すのが定石です。ping / arp は最上位（IP 層）の一手段にすぎず、そこで止めると物理・データリンクの故障を見逃します。

### 階層別 切り分けフロー

```text
PLC/DCS との通信が切れた
  ├─ ① 物理層（ケーブル・コネクタ・終端）
  │    ├─ ケーブル断線・コネクタ抜け・曲げ半径違反
  │    ├─ HUB/スイッチ/メディアコンバータの電源 ON
  │    ├─ ポートのリンク LED 点灯・速度/全二重の一致
  │    ├─ 光ファイバは受光レベル低下（コネクタ汚れ・曲げ）
  │    └─ シリアル系（RS-485）は終端抵抗 120Ω が両端のみか
  ├─ ② データリンク層（アドレス・速度・極性）
  │    ├─ ノードアドレス/局番の重複（同一バスで重複＝通信不能）
  │    ├─ ボーレート・パリティ・ストップビットの全機器統一
  │    ├─ RS-485 の A/B（+/−）極性（機器で定義が逆のことあり）
  │    └─ 通信エラーカウンタ（CRC/フレーミングエラー）の増加確認
  └─ ③ 上位層（IP・コントローラ間ネットワーク）
       ├─ IP アドレス重複・サブネット/VLAN 誤設定
       ├─ 冗長ネットワーク（Aリング/Bリング）のどちらが生きているか
       └─ ネットワーク負荷・ブロードキャストストーム
```

### ① 物理層の実測ポイント

- **リンク LED が消灯**: ケーブル・コネクタ・相手ポートを疑う。別ポート・別ケーブルへ差し替えて切り分ける
- **リンクは点灯するが通信しない**: 速度/全二重のオートネゴシエーション不一致（片側固定・片側 Auto）を確認する
- **RS-485 の終端抵抗**: バス両端の機器のみ 120Ω。中間機器に付けると信号が減衰しエラー多発。詳細は [調節計・記録計](../03-keiso/controller.md#rs-485-modbus) の Modbus RTU 設定を参照
- **光ファイバ**: コネクタ端面の汚れ・過度な曲げで受光レベルが落ちる。清掃後に再挿入して改善するか確認する

### ② データリンク層の着眼（プロトコル別）

| プロトコル系統 | 重複・整合で見る点 | 典型症状 |
|---------------|------------------|---------|
| Modbus RTU（RS-485） | 局番重複・ボーレート/パリティ不一致・A/B 極性 | 特定局だけ無応答／全局タイムアウト |
| Modbus TCP・産業 Ethernet 系 | IP 重複・ポート・コネクション数上限 | ある機器を境に応答しなくなる |
| コントローラ間ネットワーク | 冗長リングの片系断・負荷過多 | 通信は生きるが遅延・間欠断 |

!!! tip "エラーはカウンタで診る"
    多くの通信モジュールは CRC エラー・フレーミングエラー・再送回数を内部カウンタで保持しています。
    「通信は繋がるが時々切れる」の間欠障害は、このカウンタの増加ペースを見ると
    ノイズ（物理層）か設定不整合（データリンク層）かを切り分けられます。
    ノイズが疑わしければケーブル経路（動力線との近接）とシールド接地を確認します。

### ③ 上位層の確認コマンド（エンジニアリング PC から）

| 確認内容 | コマンド例 | 読み取り |
|---------|----------|---------|
| 疎通確認 | `ping [対象 IP]` | 無応答＝経路 or 相手停止／遅延大＝負荷・間欠断 |
| 接続機器一覧 | `arp -a` | 相手 MAC が出るか＝L2 到達性。出ない＝物理/L2 断 |
| 経路確認 | `tracert [対象 IP]` | どのホップで止まるか＝切断箇所の切り分け |

!!! warning "上位コマンドが通っても油断しない"
    ping が通るのに I/O 値が更新されない場合、切れているのは IP ネットワークではなく
    「コントローラ ↔ I/O 間の制御ネットワーク」や「スキャン設定」です。
    上位の疎通確認は通信断調査の入口であって、結論ではありません。

!!! tip "通信ハードのフリーズは電源再投入で解ける場合がある"
    通信モジュール内部のフリーズは電源再投入で解消することがあります。
    ただし再起動前に「再起動の影響範囲（どのプロセスが止まるか）」を製造担当に確認してください。

## I/O 点が読めない

「現物 → カード LED → ソフト設定」の順に突き合わせます。1点だけか、カード単位か、ラック単位かで原因が大きく分かれます。

### まず症状で当たりを付ける

| 症状 | 有力な原因 |
|-----|----------|
| 特定の1点だけ読めない | チャネル配線の断線・短絡、フィールド機器故障、チャネルヒューズ断 |
| カード単位で全点読めない | I/O カード故障・カード電源/ヒューズ断・バス接続不良 |
| ラック単位で全点が 0 | ラック電源喪失・ベースユニット/バックプレーン異常 |
| 値が固定のまま変化しない | スキャン停止・強制（フォーシング）残置・ソフト設定ミス |

### モジュールステータス LED の読み方

I/O カードの LED は「モジュール全体」と「チャネル個別」の2階層で見ます（色・表記は機種のマニュアル参照。以下は一般的な意味）。

| LED 種別 | 一般的な意味 | 判断 |
|---------|------------|------|
| RUN / MOD（緑） | モジュール正常稼働 | 消灯＝カード電源喪失・CPU からの認識不良 |
| ERR / FLT（赤・橙） | モジュール異常・診断エラー | 点灯＝カード故障・設定不整合・フィールド異常 |
| I/O・チャネル LED（緑） | 各チャネルの ON 状態 | DI/DO は現場状態と一致するか突き合わせる |
| チャネル異常 LED（赤・橙） | チャネル単位の断線・短絡・レンジ外 | 点灯チャネルを特定して端子を実測 |

!!! tip "チャネル単位の断線・短絡検出（オープン検出／NAMUR）"
    診断機能付き AI カードは、4-20 mA のフェイル判定（NAMUR NE43）でチャネルごとに
    断線・センサ異常を検出します。判定電流のしきい値は
    [計装の基礎](../03-keiso/basics.md) の 4-20 mA フェイル判定を正典として参照してください（本ページでは数値を再掲しません）。
    DI カードでも NAMUR タイプ（IEC 60947-5-6）は微小電流を常時流し、
    配線の**断線＝ほぼ 0 に近い電流／短絡＝過大電流**を接点 ON/OFF とは別に検出します。
    ラインフォールト（LFD）表示が出たチャネルは、まず端子を実測します。

### フィールド側端子電圧・電流の実測切り分け

カード交換の前に、フィールド端子で実測して「異常はカードか・配線か・現場機器か」を確定させます。

- **アナログ入力（4-20 mA）**: 端子でループ電流をテスタ（電流レンジ）または直列のループチェッカーで実測する
    - 4 mA 未満・0 mA 付近 → 断線・電源喪失側。伝送器の供給電圧（一般に DC24 V 系）も確認
    - 20 mA 超・振り切り → 短絡・伝送器ハイ側故障
    - 実電流は正常なのに DC 表示だけおかしい → カードの入力回路 or スケール/レンジ設定を疑う
- **デジタル入力（接点）**: 端子間電圧を電圧レンジで実測する
    - 接点 ON のはずなのに入力電圧が出ない → 接点不良・配線断・電源喪失
    - 電圧は出ているのに DI が OFF のまま → 入力回路故障 or しきい値/極性（シンク/ソース）設定違い
- **ヒューズ**: I/O カード専用ヒューズ・電源分配盤ヒューズの導通を確認する

!!! warning "実測は活線。感電と誤トリップに注意"
    通電状態での端子測定は感電リスクがあり、テスタの誤接触が誤入力・誤出力を招きます。
    出力（DO/AO）側の測定は現場機器を不用意に動かすため、製造担当への事前連絡と
    測定レンジ（電流/電圧）の取り違え防止を徹底してください。

## 冗長化系の切り分け

化学プラントの DCS/PLC はコントローラ・電源・通信を冗長化しているのが基本です（構成は [DCS](../03-keiso/dcs.md#_2) の冗長化の考え方を参照）。冗長系のトラブルは、**まず「今どちらが現用（アクティブ）か」を確定**させることから始めます。

### 現用系・待機系の判定

| 確認対象 | 見る場所 | 正常な状態 |
|---------|---------|-----------|
| 制御 CPU | CPU 前面の ACT/STBY LED・エンジニアリングツールの系表示 | 一方が ACT、他方が STBY で健全 |
| 電源 | 各電源ユニットの出力 LED・電圧監視 | 両系とも正常出力（片系でも運転継続可） |
| 通信 | 冗長リング/バスの A系・B系のリンク状態 | 両系リンク、片系断でも縮退で継続 |

!!! danger "片系故障＝冗長喪失状態。放置しない"
    片系が故障しても運転は継続しますが、その間は**残る1系の単一故障で全停止**します。
    片系故障アラームは「まだ動いているから後で」ではなく、冗長を回復するまでの
    リスクが上がった状態として、速やかに原因調査・復旧に入ってください。

### 手動切替の可否と注意（バンプ）

- 正常な冗長系は待機系が現用系と同期しており、切替は**バンプレス（無瞬断）**が原則です
- ただし待機系の同期が取れていない・待機系自体が異常な状態で手動切替すると、
  出力が飛ぶ「**バンプ**」が起きてプロセスが乱れる恐れがあります
- 手動切替の前に必ず「待機系が健全・同期済みであること」を確認し、製造担当に切替タイミングと影響を通知してから実施します

### 冗長 CPU 切替手順（一般手順）

1. 待機系 CPU が正常・同期済みであることを確認（ここを飛ばさない）
2. 製造担当へ切替を通知し、実施許可を得る
3. 現用系 → 待機系へ切替（エンジニアリングツール操作 or ハードスイッチ）
4. 切替後、全 I/O・全ループが正常に追従しているか確認
5. 異常系を切り離して原因調査（部品交換・メーカーサポートは切り分け後）

!!! note "縮退運転中の影響範囲を把握しておく"
    冗長化が省かれている I/O カードや単一系のループは、故障時に対象ループが全て手動になります。
    どのループが縮退で手動化するかを平常時に P&ID・I/O リストで押さえておくと、
    片系故障時の運転制限判断が速くなります。

## プログラム・ロジックが意図通り動かない

「起動ボタンを押しても動かない」「指令を出しても弁が動かない」の多くは、**インターロック（起動許可）条件の不成立**です。オンラインモニタで条件を1つずつトレースします。

### まず切り分ける

- **入力は正常・出力は異常** → ラダー/ロジック側の問題（条件不成立・強制残置）
- **入力が異常** → フィールド信号・配線の問題（前章の実測へ）
- **出力は出ているが現場機器が動かない** → 出力先の配線・機器・電源の問題

### オンラインモニタでインターロック条件をトレースする

1. ラダー/ファンクションブロックをオンラインモニタで開く
2. 起動許可条件（AND に入る接点・比較・タイマ）を1つずつリスト化する
3. 各条件の現在値を確認し、**成立していない条件を特定**する
4. その条件の入力信号について、現場の物理値と DCS 表示が一致するか突き合わせる
5. タイマ/自己保持（ラッチ）が絡む場合は、いつセット/リセットされたかをトレンドで追う

!!! tip "タイムチャートで動作を追う"
    トレンド機能でインターロック入力信号と出力指令を同時に記録し、
    「何時何分に何が変化したか」を並べると、原因の時系列が一目で分かります。

### 強制（フォーシング）の危険と痕跡確認

!!! danger "フォーシングは安全機能を殺す。必ず痕跡を確認する"
    試運転や切り分けで入出力を強制（フォース）した後、**解除し忘れ**が最も危険な残置です。
    「値が固定のまま変化しない」症状では、まず**強制の有無**を確認してください。
    多くのツールは強制中の点を一覧表示・画面上でマーキングします。
    強制は原則2名確認・時間管理・解除確認をセットにし、SIS 関連の強制はさらに厳格に扱います。

### スキャンタイム超過の兆候

- ロジックが「時々だけ遅れる・追従が鈍い」場合、スキャンタイム（周期）の超過を疑います
- ウォッチドッグタイマ超過アラーム、スキャンタイムの実測値が設定周期に迫る、
  プログラム増設・通信負荷増加の直後に発生、などが兆候です
- 割り込み処理・通信処理の増加が周期を圧迫していないかを、ツールのスキャンタイム表示で確認します

## アラーム氾濫（alarm flood）と SOE

トリップ時に何十・何百のアラームが一斉に出ると、**本当の初発事象（1点）が埋もれます**。氾濫したアラーム群は「結果」であり、原因は最初に起きた1点です。

### 初発事象（first-out）の見つけ方

- **SOE（Sequence of Events：事象記録）** を使います。SOE は各接点の変化をミリ秒精度のタイムスタンプで記録するため、**最も早い時刻の事象＝初発事象**が特定できます
- 通常のアラーム履歴（秒精度・表示遅延あり）では前後関係が潰れることがあるため、原因追跡には SOE の時系列を優先します
- 多くのトリップ系には「ファーストアウト（first-out）表示」機能があり、最初にトリップ条件を満たした点を保持します。これをまず読みます

### ラッチ（自己保持）の切り分け

- トリップ出力が保持（ラッチ）される設計では、**原因が消えてもアラームは消えません**。「現場は正常に戻ったのにトリップが解けない」はラッチ残りが typical です
- リセット操作でラッチを解除する前に、**初発事象が本当に復帰しているか**を確認します。原因未解決のままリセットすると再トリップします
- 自己保持回路（セット優先/リセット優先）のどちらかで挙動が変わるため、ロジック上のセット/リセット条件をオンラインモニタで確認します

!!! tip "氾濫時はまず時刻の早い順に並べ替える"
    アラーム一覧を発生時刻の昇順に並べ、SOE の最初の数点だけを見ます。
    後続の大量アラームは初発事象の波及なので、初発1点の復旧に集中すれば芋づる式に解けます。

## バッテリー低下アラームの対応

!!! warning "バッテリー低下アラームは早急に対応する"
    バッテリーが完全放電すると、停電復帰後にプログラムが消失する機種があります。
    アラーム発生後 1〜2 週間以内の交換を推奨します（機種別の交換周期目安は
    [PLC 基礎](../02-teiatsu/plc.md#_7) を参照）。

### バッテリー交換手順

1. **事前準備**
    - バッテリー品番を確認（機種・メーカーにより異なる）
    - プログラム・パラメータを PC にバックアップ
    - 通電状態での交換可否と、無通電で保持できる猶予時間を機種マニュアルで確認
2. **交換作業**
    - 原則は通電状態のまま交換（電源を切るとプログラムが消える機種あり）
    - 交換猶予時間内（通常数分）で完了させる
3. **交換後確認**
    - プログラムが正常動作しているか確認
    - バッテリーアラームが消灯したか確認

## 保守時の鉄則（活線でのカード抜き差し）

!!! danger "ホットスワップ対応を確認せずにカードを抜かない"
    活線挿抜（ホットスワップ）に対応していないカードを通電中に抜くと、
    ラック全体のリセット・他チャネルの瞬断・機器破損を招きます。
    抜き差しの前に、そのカードが**ホットスワップ対応か**を機種マニュアルで必ず確認してください。

- 対応機種でも、抜く前に対象チャネルの**影響範囲（どのループ・インターロックが落ちるか）を製造担当と確認**します
- SIS のカード・SIS 連携 I/O は活線作業でも安全機能のバイパス扱いになり得ます。SIS 絡みの作業は [機能安全（SIS/SIL）](../03-keiso/sis-sil.md) のバイパス管理（2名確認・時間管理・見える化）に従い、SIS 担当・安全担当を関与させます
- カード交換後は設定・レンジ・アドレスが元と一致しているかを点対点で確認します

## 定修での DCS 設定変更手順（MOC）

!!! warning "DCS/PLC の設定変更は必ず MOC（変更管理）手続きを踏む"
    製造プロセスに影響する設定変更は MOC（Management of Change）が必要です。
    「少し変えるだけだから」で MOC を省略すると重大事故の原因になります。
    MOC の詳細フローは [DCS](../03-keiso/dcs.md#moc-management-of-change) を参照してください。

### 変更後の確認事項

- [ ] 変更した設定が意図通りに動作しているか
- [ ] 関連するインターロックが正常に機能しているか
- [ ] 強制（フォース）を全て解除したか
- [ ] DCS/PLC の設定バックアップを更新したか
- [ ] 図面（計装図・制御ロジック図）を最新版に更新したか

## SIS との違い

!!! danger "DCS と SIS は独立したシステム。混同しない"

| 項目 | DCS | SIS（安全計装システム） |
|------|-----|---------------------|
| 目的 | プロセス制御 | 安全機能（緊急遮断） |
| 故障時の動作 | 可用性を優先 | 安全側（フェールセーフ）を優先 |
| 変更管理 | MOC | SIL 評価を含む厳格な変更管理 |
| テスト | 通常運転中に確認可能 | 定期的なプルーフテスト（機能テスト） |
| 電源 | 共通電源から供給可 | 独立した電源系統 |

SIS の設定変更・信号確認・強制・カード作業には DCS とは別の手続きが必要です。SIL・投票構成・プルーフテスト・バイパス管理の実務は [機能安全（SIS/SIL）](../03-keiso/sis-sil.md) を参照し、SIS に関わる作業は SIS 担当者・安全担当者を必ず関与させてください。
